    iOS架构设计方案

    wanyakun 4/10/2017 iOS

    # 现状

    • 公司好几个app都是从一款app基础上开发的，Comms模块可以分步提出做成基础Library
    • 业务模块划分混乱
    • 业务开发模式简单，MVC模式
    • 工程混乱
    • 代码不规范，命名不规范

    # 架构设计

    基于公司目前几款app的现状，设计了iOS整体的架构方案和App开发的架构方案。

    # 升级开发模式

    # 组件化

    组件是可以独立开发，独立发布，独立运行的。具备独立性的组件可以很好的隔离跨团队之间的依赖，彼此独立开发，按照各自的节奏发布版本

    每个独立的组件对发布的版本号负责，不论是其他组件还是主工程都依赖组件发布的稳定版本。

    • 引入依赖管理设施CocoaPods
    • 基础平台Framework化
    • 业务模块Framework化

    # 独立开发

    剥离基础平台中各个模块，剥离业务模块，将剥离后的内容独立工程化，做成CocoaPods库。需要建立私有Specs。

    • 基础平台各个模块剥离成独立的工程开发
    • 业务模块剥离成独立的工程开发

    # 独立发布

    让每一个独立组件尽可能少的依赖其他组件，在最小范围内正常编译执行。每次发布不再是一个稳定版本号，而是一个稳定的二进制包。

    • 梳理组件之间的依赖关系
    • 尽可能减少组件间的依赖

    # 执行方法

    整个模式升级基本上经历这样几个阶段：

    • 代码独立，先从形式上解耦
    • 独立代码工程化，为独立运行打下基础
    • 梳理依赖关系，独立工程可编译
    • 放弃源码依赖，提速集成编译

    # 依赖关系梳理

    一个工程，从原始状态经过几年的开发迭代，依赖关系会变的非常复杂。需要仔细梳理。

    我们要做的并不是把这些依赖和耦合一一处理掉，而是进行梳理，把不合理的找出来，解决掉，让整个工程处在一个健康合理的依赖和耦合范围内。有问题的依赖基本有这样几种：

    • 模块循环依赖
    • 层间反向依赖
    • 非强功能依赖

    # 耦合与解耦

    耦合可以大致分为三类：

    • 界面耦合

    就是用户操作流程里，从首页-到详情-到购买页-再到充值，这些界面的跳转是硬编码的

    • 依赖耦合

    顾名思义，两个模块之间的有依赖，就是耦合

    • 工程耦合

    每个模块有自己的生命周期和运行时，每个模块在生产环境里又需要依赖主工程的运行时

    那么针对以上三种不同的耦合就需要有三种不同的解耦方案

    # 界面解耦

    界面耦合主要是通过统跳协议 & Rewrite引擎来进行解耦，也就是Router

    # Rewrite引擎

    首先，无论是Native还是Web，在App中他们两两之间的跳转都被统跳协议托管，而统跳协议在执行跳转操作之前会把原始URL放入Rewrite引擎中做一次Rewrite操作。这样一来，Rewrite引擎就根据配置规则，把原始URL转换成适用于App的目标URL，实现了URL表现平台一致性。

    Rewrite引擎模拟Web容器（Apache/Nginx等）的Rewrite配置，根据配置把传入的原始URL进行重写，返回重写后的目标URL，交给统跳协议处理。

    配置是通过正则表达式描述的Rewrite规则列表，这份列表通过App的配置中心实现动态更新。

    # 依赖解耦和工程解耦

    BeeHive是一个运行时框架，主要解决依赖耦合和工程耦合。

    BeeHive（BeeHive已经开源，可以在Github上看到源码：https://github.com/alibaba/BeeHive）

    BeeHive的原理是，每一个对外提供服务的模块，需要注册一个抽象接口到Beehive提供的Interfaces（接口池）。注意，在这个池子里只有抽象接口。

    开发阶段，调用方依赖接口池中响应的接口，并以接口为参数，通过BeeHive提供的工厂方法获取一个服务实例，这个实例可以正常进行服务。

    运行时阶段，BeeHive工厂方法根据服务的注册配置，构造服务实例。若：当前的运行环境没有依赖提供服务的模块，则返回空；若：当前运行环境依赖关系完整，则开始构造服务，并返回。

    # BeeHive和统跳&Rewrite的区别

    BeeHive和统跳协议的目的都是解耦，然后二者所关注的重心不同。

    • 统跳主要为界面解耦服务，业务要求界面链路的强动态性；
    • Beehive则为模块解耦，解决模块强依赖带来的开发阶段痛苦。
